Guia de Autenticação com Clerk

Este documento explica tudo o que configuramos até agora no seu projeto Next.js e como as funções do Clerk operam para simplificar todo o processo de autenticação.

1. Como a integração inicial funcionou

A grande vantagem do Clerk é reduzir drasticamente a quantidade de código clichê (formulários, validação, controle de sessão). Ao rodar o comando clerk init, ele aplicou os três pilares iniciais no seu projeto:

  • Instalação dos SDKs: Instalou o pacote @clerk/nextjs, que contém tudo que é necessário (componentes visuais e lógicas de sessão).
  • Chaves de Ambiente (.env.local): Gerou e conectou automaticamente o projeto local com a sua aplicação na nuvem do Clerk, guardando chaves de segurança para autenticar o servidor.
  • Configuração Global: Modificou seu layout.tsx para garantir que o projeto inteiro pudesse ler o status do usuário.

A Magia das Chaves no .env.local

O arquivo .env.local é o coração da comunicação entre o seu projeto e os servidores do Clerk. Ele guarda as chaves que provam que o seu site tem permissão para acessar o seu painel de usuários na nuvem.

As Duas Chaves Principais (Obrigatórias em Qualquer Projeto)

  • NEXT_PUBLIC_CLERK_PUBLISHABLE_KEY: Como o nome diz (Public), esta chave é segura para ser enviada ao navegador do usuário. Ela permite que a interface (os botões de login, modais) converse com o Clerk para autenticar a pessoa na tela.
  • CLERK_SECRET_KEY: Nunca pode vazar! Ela fica escondida no servidor do seu projeto. É com ela que o seu backend garante que as requisições que chegam de fato são válidas, e permite fazer ações administrativas (como buscar dados no banco com o auth()).

Projetos Iniciantes vs Avançados

A base de segurança é exatamente a mesma para um projeto júnior ou para um site com 1 milhão de acessos. O Clerk escala de forma transparente. No entanto, projetos mais avançados costumam adicionar algumas chaves extras:

  • URLs Customizadas: Em vez de usar os modais flutuantes, projetos grandes criam páginas próprias (ex: /entrar, /registrar). O Clerk usa variáveis como NEXT_PUBLIC_CLERK_SIGN_IN_URL=/entrar para saber para onde direcionar as pessoas.
  • Webhooks: Quando um usuário se cadastra, o Clerk pode avisar o seu banco de dados de forma silenciosa no fundo. Para garantir que esse aviso seja seguro, usa-se uma chave extra como WEBHOOK_SECRET.

2. O Arquivo proxy.ts (Proteção de Rotas)

O arquivo proxy.ts atua como um "guarda-costas" da sua aplicação.

  • config.matcher: É a lista de rotas que o guarda-costas precisa vigiar. Adicionamos a rota /__clerk/:path* lá, permitindo que o Clerk processe internamente a verificação de token de forma muito rápida.
  • clerkMiddleware(): Permite definir quais rotas são abertas e quais exigem login. Usamos o auth.protect() dentro dele para barrar acessos não autenticados.

3. Componentes de Interface Utilizados

O Clerk possui diversos componentes prontos que poupam muito tempo de desenvolvimento:

<ClerkProvider>

Onde foi usado: No app/layout.tsx.

O que faz: Empacota toda a aplicação. Gerencia o estado global de autenticação e distribui as informações do usuário. Sem ele, os outros componentes não funcionam.

<Show>

Onde foi usado: No app/page.tsx e layout.tsx.

O que faz: Controla a renderização condicional para não termos telas piscando ao carregar o estado do usuário.

  • <Show when="signed-in">: Aparece apenas se a pessoa estiver logada.
  • <Show when="signed-out">: Aparece apenas para visitantes deslogados.

<SignInButton> e <SignUpButton>

Onde foi usado: No layout.tsx e app/page.tsx.

O que faz: Envelopam botões comuns para acionar o fluxo de login de forma segura, abrindo modais nativos do Clerk automaticamente.

<UserButton>

Onde foi usado: No layout.tsx e app/dashboard/page.tsx.

O que faz: Renderiza a foto do usuário e abre uma central completa para gerenciar a conta (alterar senha, segurança e botão de Sair).

4. O que mais o Clerk pode fazer?

💡 Buscar informações no Backend (auth)

Nos seus Server Components do Next.js, você pode chamar const { userId } = await auth(); para saber com total segurança qual usuário está fazendo a requisição antes de bater num banco de dados.

📝 Componente <UserProfile />

Caso queira uma página /perfil dedicada, basta usar este componente na tela. Ele renderiza a central inteira de configurações de conta na sua própria página.

🏢 Organizações (B2B)

Se o seu projeto for um SaaS para empresas, o Clerk permite ativar Organizações, onde usuários criam times e gerenciam permissões de membros. Isso é feito usando o componente <OrganizationSwitcher />.
